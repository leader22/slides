MobXのコンセプト

Automatic derivation

Anything that can be derived from the application state, should be derived. Automatically.

意訳: アプリの状態に関するすべての事柄は、その状態変更に呼応して自動的に成されるべき。

昨今のアーキテクチャのトレンド

  • 関心の分離
  • 一方向のデータフロー
    • view = fn(state)
    • どのライブラリもだいたいそう

Stateを変更（Action）さえすれば、後はMobXが自動的にやってくれる。

どんなコードになるか

必要なのは3ステップ

  • 1. Stateを定義すること
  • 2. そのStateを使って関数を実行すること
    • view.render()とか
  • 3. Stateを変更すること

1. ObservableなStateを用意

const { observable } = require('mobx');

const state = observable({
  count: 1,
});

https://mobx.js.org/refguide/api.html#creating-observables

※RxとかのObservableではないのでご注意。

2. Stateでなんかする

const { autorun } = require('mobx');

autorun(() => {
  // いわゆる`render()`的なこと
  $el.textContent = state.count;
});

https://mobx.js.org/refguide/autorun.html

3. Stateを更新

state.count = 2;

これだけで、自動的にさっきの関数が呼ばれます。

1 + 2 + 3なコード

const { autorun, observable } = require('mobx');

// state
const state = observable({
  count: 1,
});

// view
autorun(() => {
  console.log(`Count is ${state.count}!`); // Count is 1!
});

// action
state.count = 2; // Count is 2!
state.count = 3; // Count is 3!

ね、簡単でしょ？

おどろきのSimpleさ

  • State: mobx.observable()
  • Action?: Stateに対する変更（任意
  • View?: Stateの変更に対するリアクション（任意

ViewやActionの粒度は開発者次第、MobXは何も規定しない。

もう少しだけコード

State: Classで書く

class Store {
  constructor() {
    extendObservable(this, {
      items: [],
    }),
  }
}

or

class Store {
  @observable items = [];
}

Decoratorsの使用はマストではないが、使えるなら使いたくなると思う・・！

State: Computed value

class Ticket {
  @observable price = 1000;
  @observable amount = 1;

  @computed get total() {
    return this.price * this.amount;
  }
}
const ticket = new Ticket();
console.log(ticket.total); // 1000
ticket.amount = 2;
console.log(ticket.total); // 2000

特定の状態に由来する状態をcomputedで宣言的に。

  • いちいち手動で更新しなくていいので事故らない
  • 扱うべきものに集中できるので、stateの概観がブレない

View: Reactするなら

const React = require('react');
const { observer } = require('mobx-react');

const View = ({ todoStore }) => (
  <div>
    残TODO数: {todoStore.unfinishedCount}
    <ul>
      { todoStore.items.map(todo => (
      <li key={todo.id}
        <Todo item={todo} />
      </li>
      )) }
    </ul>
  </div>
);

module.exports = observer(View);

observerはさっきのautorunのラッパー。 これで必要なstateに更新があった時だけ、自動でrender()されるように。

View: 以外の副作用も

// for rendering views
ReactDOM.render(<View todoStore={store} />, $el);
// for others
reaction(
  () => store.unfinishedCount,
  count => {
    if (count > 10) {
      askColleagueToHelp(store.items);
    }
    if (count > 100) {
      reportToBoss(store.items);
    }
  }
)

observableな値の変化に対するreactionを指定することで、Viewの表示に関することも、それ以外のことも、State由来の事柄は全てを宣言的・リアクティブに書ける。

Action: 変更に制限を

const { useStrict, action } = requre('mobx');

useStrict(true);

state.count = 10; // Error!
const modCount = action(() => {
  state.count = 10;
});

modCount(); // Ok!

action()でラップすることで、その関数からしかStateが変更できないようにできる。

Action: 非同期もそのまま

uiStore.setLoading(true);

fetch('/api/my/todos/')
  .then(items => {
    todoStore.init(items);
    uiStore.setLoading(false);
  })
  .catch(err => {
    uiStore.showError(err);
  });

非同期だからといって特殊なことをする必要はない。 直感的に、ただStateを更新するだけ。

最低限のコードの組合せ

// Store(State + Action)
class TodoStore {
  @observable items = [];

  @action
  addTodo(title) {
    this.items.push(new Todo(title));
  }
}
<!-- View -->
<input type="text" /><button onClick={addTodo}>ADD</button>
// Event
function addToto() {
  const title = $input.value;
  store.addTodo(title);
}

※コードはイメージです

お気持ち

個人的には「推し」です

  • 「顧客が本当に必要だったもの」感
    • 書き味・取り回しの軽さ
  • コード・ファイルが少ない = 楽
  • それなりのものを、それなりに作れる
    • 最低限の部品とその役割分担が妥当

アーキテクチャを規定しない

  • このライブラリではどう書けばいい・・？がない
    • 本当に無駄な時間だと思う
  • 他人のルールに「無理やり納得」しなくていい
    • ○○uxのActionオブジェクトは未だに謎
      • 一方向フローは良いとしてもtypeて
      • タイムトラベルとかしないし・・
  • ある程度の自由度を許容した「遊びをもたせた設計」もできる

隙のないカチカチな設計が綺麗に運用されてるの見たことない・・

なのでもちろん

  • 最低限の線引はマスト
    • 己の力量が試される
    • 自信がないなら、Opinionatedなものを
    • エンジニアの本来の仕事なんですけど・・
  • どんなライブラリも慣れは必要
